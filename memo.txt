commande initiale
-----------------
Sans toucher au code existant, car il a été développé par une autre équipe et je souhaite le garder intact, je souhaite développer une webapp qui me permette d'étendre les fonctionnalités de Hathor en VR. Celle-ci se trouvera dans le dossier wapps. Pour résumer, je souhaite créer une application de visite virtuelle guidée du site de Landevennec, 
sur lequel je dispose d'un 3D tiles de l'état existant, auquel on superposera plusieurs fichiers GLTF de restitution 3D des espaces manquants, ainsi que quelques objects 3D generés par Three.js (essentiellement des plans orientés dans l'espace et donnant à voire des photographies d'archives recalées dans l'espace 3d).
Cette application sera déployée sur un NUC, en local, sur lequel seront connectés les casques de réalité virtuelle des visiteurs via un router WIFI. Ceux-ci seront guidés par un médiateur / coférencier qui utilisera soit une tablette sous android, soit un casque de VR, pour contrôler la visite.
Ce contrôle implique donc que nous aurons un utilisateur "maître", qui pourra décider du rythme de la visite, des points de vue à adopter, de l'affichage des différents calques (état actuel / état XIIIe siècle), et d'un certain nombre d'utilisateurs "visiteurs" disposant de fonctionnalités limitées. 
Ils pourront par exemple se déplacer dans l'espace immersif, activer des zones sensibles affichant des informations, mais ne pourront pas imposer les points de vue aux autres utilisateurs ou gérer l'affichage des différents calques.
Pour cette application VR, nous utiliserons les dispositifs VR Meta Quest 3, incluant comme outils de contrôle les manettes ainsi que les mains (pincement de doigts, mouvement de pouce, etc); cette dernière modalité sera d'ailleurs préférables pour des raisons logistiques (perte de manette, problème de batteries, etc).
L'ancienne version de la plateforme Aton proposait notamment un certain nombre d'exemples de gestion de visites multi-utilisateurs, qui n'ont pas encore été adaptés à la nouvelle version. Ceux-ci se trouvent dans public/old/examples.
Peux-tu dans un premier temps étudier ces exemples, comprendre le fonctionnement des features qui correspondent à mon objectif de créer une visite virtuelle multi-utilisateur, en m'en faire une synthèse. Nous verrons ultérieurement comment les adapter à mes besoins et les transposer dans la nouvelle version de la plateforme.

Controles
---------
Reprenons la mise en place des fonctionnalités de contrôles

Le médiateur peut controler la session de visite avec une tablette ou un casque VR. Sur tablette/smartphone, il a accès à une interface sans rendu 3D (déjà realisée, voir dans /wapps/landevennec/control.html). Sur tablette/smartphone, il a également accès à une interface 3D, où il participe pleinement à la visite (avatar), mais avec les fonctionnalités étendues à celles présentes dans la version sans rendu 3D.
Outre le contrôle de l'affichage des calques, des points de vue prédéfinis de la visite, et du partage de sa vue en cours, il peut également décider d'activer/désactiver les possibilités de navigation des visiteurs (téléportation) afin de maintenir la cohésion du groupe.

En mode VR, le visiteur doit pouvoir observer la scène en se mouvant dans l'espace avec son casque. Il doit pouvoir également faire des rotations incrémentales de x degré (par defaut, disons 15°) vers sa gauche ou vers sa droite, à l'aide de sa main gauche, ou de deux boutons de la manette gauche. Par défaut, il ne peut pas se téléporter (gachette index droit ou pincement index/pouce main droite) pour éviter de se détacher du groupe. Le médiateur peut activer cette fonctionnalité à un moment du parcours lorsqu'il souhaitera proposer une visite libre.

Il faudra également rendre possible l'affichage du contenu des annotations sémantiques (du texte et quelques images)

Avant de réaliser l'ensemble des features, propose moi un plan de developpement, en essayant si possible, et si cela est pertinent, de fractionner le main.js en modules fonctionnels cohérents.

Solution immédiate — mkcert (setup une fois par casque)
  
  mkcert crée une autorité de certification locale que tu fais confiance sur chaque appareil. Une fois installée, aucun avertissement — le cadenas vert apparaît comme sur n'importe quel site public.

  Sur le NUC :

  # Installer mkcert (Ubuntu/Debian)
  sudo apt install libnss3-tools
  curl -JLO https://github.com/FiloSottile/mkcert/releases/latest/download/mkcert-v*-linux-amd64
  chmod +x mkcert-* && sudo mv mkcert-* /usr/local/bin/mkcert

  # Créer l'autorité de certification locale
  mkcert -install

  # Générer le certificat pour l'IP du NUC
  mkcert -key-file config/certs/server.key \
         -cert-file config/certs/server.crt \
         192.168.1.18 localhost

  # Récupérer le fichier CA à distribuer aux casques
  cp "$(mkcert -CAROOT)/rootCA.pem" /var/www/html/rootCA.pem
  # (ou tout autre répertoire servi par ATON — public/ par exemple)

  Sur chaque Meta Quest, une fois :

  1. Dans le navigateur du casque, ouvrir http://192.168.1.18:8080/rootCA.pem
  2. Le fichier se télécharge
  3. Paramètres → Wi-Fi & Internet → (icône ⚙ en haut) → Certificats → Installer un certificat CA
  4. Sélectionner le fichier téléchargé → Confirmer

  Ensuite https://192.168.1.18:8083 s'ouvre sans avertissement sur ces casques.

  ---
  Solution pérenne — nom de domaine + Let's Encrypt (recommandé pour la prod)

  C'est la vraie solution propre, sans rien à installer sur les casques.

  Principe : un nom de domaine public pointe vers l'IP privée du NUC. Let's Encrypt délivre un certificat valide. Les casques résolvent le domaine → IP locale et voient un cert de confiance universel.

  DNS public :  visite.abbaye-landevennec.fr  A  192.168.1.18
  Let's Encrypt : cert valide pour ce domaine
  Quest : résout visite.abbaye-landevennec.fr → 192.168.1.18 → HTTPS ✓

  Étapes :

  1. Enregistrer un domaine — ~10 €/an (OVH, Gandi…). Peut être un sous-domaine si l'abbaye a déjà un domaine.
  2. Créer l'enregistrement DNS :
  visite.abbaye-landevennec.fr  A  192.168.1.18
  2. (oui, une IP privée dans un enregistrement DNS public — c'est valide et courant)
  3. Obtenir le certificat via DNS-01 (pas besoin d'ouvrir le NUC sur internet) :
  sudo apt install certbot
  sudo certbot certonly --manual --preferred-challenges dns \
    -d visite.abbaye-landevennec.fr
  # certbot demande d'ajouter un enregistrement TXT chez votre registrar
  # → coller la valeur dans l'interface DNS, valider
  4. Installer dans ATON :
  sudo cp /etc/letsencrypt/live/visite.abbaye-landevennec.fr/privkey.pem \
          config/certs/server.key
  sudo cp /etc/letsencrypt/live/visite.abbaye-landevennec.fr/fullchain.pem \
          config/certs/server.crt
  5. Les casques accèdent à https://visite.abbaye-landevennec.fr:8083 — cadenas vert, aucune manipulation préalable.
